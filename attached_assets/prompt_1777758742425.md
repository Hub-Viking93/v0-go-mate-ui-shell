# PHASE 4 E2E SPINE TEST — comprehensive frontend-driven verification

This is THE test that proves the entire Spine works end-to-end as a real user experiences it. After Phase 4 ships (4.1 dashboard layout + 6.1 guide composer + 6.2 citations + 4.3 audit popover), this test verifies the system delivers exactly what the spec promises — for four distinct user personas, on real infrastructure, with no shortcuts.

Write `artifacts/gomate/e2e/phase-4-spine.spec.ts` using Playwright.

---

## HARD RULES — violating any of these invalidates the test

### 1. Frontend-only execution

The test runs against the live deployed app via Playwright. The test:
- Clicks buttons via `page.click()`
- Types into inputs via `page.fill()` / `page.type()`
- Waits for network responses via `page.waitForResponse()`
- Navigates via UI links, never via direct URL manipulation outside the user flow
- Reads DOM state, never internal app state

The test does NOT:
- Write directly to Supabase mid-flow
- Mock or stub LLM responses
- Intercept or modify API requests
- Bypass auth via cookies or localStorage manipulation
- Manipulate the api-server or its memory
- Read from internal frontend state stores (Zustand, React state) — only DOM

### 2. Real everything

Every external dependency is real:
- Real Supabase auth via either the anonymous-bypass landing CTA OR the actual `/auth/sign-up` page (test alternates — see Auth Mode section below)
- Real Anthropic API calls (Claude Sonnet 4.6 for agents, Haiku 4.5 for extractor)
- Real Firecrawl scraping of official sources
- Real Server-Sent Events from `/api/chat` and research endpoints
- Real database writes via the api-server (never direct from test)

### 3. Cleanup is allowed, but only in afterEach

After each persona test completes (pass or fail), the cleanup hook may use the Supabase service role key to delete:
- The test user's `relocation_plans` rows
- Their `chat_messages`, `agent_audit`, `agent_run_log`, `artifacts` rows
- Finally, the `auth.users` row itself (works for both anonymous and permanent users)

This cleanup happens ONLY in `afterEach`, never during the test flow. The test must complete its assertions on the live state before cleanup runs.

### 3b. Auth modes tested

Each persona is tested under TWO auth modes (run as separate test cases):

**Anonymous mode (judge-experience):**
- Visit `/` (landing page)
- Click "Start your relocation plan" → triggers `signInAnonymously()`
- No email entered, no password — direct into onboarding
- Anonymous session JWT in cookie/localStorage
- After 5 fields populated, save-progress modal appears
- Test asserts modal appears at the right moment, then DISMISSES it (clicks "Continue without saving") to continue testing the anonymous flow through to dashboard

**Permanent signup mode:**
- Visit `/auth/sign-up` directly
- Enter unique test email + password
- Verify email via test-fixture extraction (read magic link from Inbucket in dev, or set `email_confirmed_at` directly in afterEach setup)
- Land on /onboarding
- Run same persona script

Each persona must pass ALL assertions under BOTH modes. Anonymous mode is the buildathon-judge-realistic path; permanent signup is the long-term-user path. Both must work.

### 4. Screenshots are evidence

Save full-page screenshots (not viewport-only) to:
```
artifacts/screenshots/phase-4/{persona-slug}/{NN}-{description}.png
```

Minimum 12 screenshots per persona:
- `01-landing` — landing page on first arrival
- `02-signup-form` — auth signup form filled
- `03-onboarding-start` — mascot in waving state, first question visible
- `04-after-bundled-input` — mascot in thinking state after first message submitted
- `05-side-panel-populated` — extracted fields visible in side panel with audit icons

- `11-audit-popover-open` — audit popover open showing chain of evidence on a profile field
- `12-citation-click` — guide page with citation footnote clicked/highlighted

### 5. No workarounds when bugs are found

The bug-handling protocol at the end of this document is binding. Read it before starting.

## TEST SCOPE — what this verifies

After Phase 4 completes, the system must deliver:

1. A new user can sign up, complete mascot-driven onboarding, watch their profile build live in a side panel, click "Generate my plan", watch a multi-agent research team work in real time, land on a profile-aware dashboard with research-grounded cards, click audit icons to see chain-of-evidence, and read a coherent personalized guide with inline citations linking to real official sources.

2. The system behaves correctly for **four distinct personas** (different citizenship, destination, family, purpose). Different personas trigger different specialists and produce different dashboard cards and different guide content.

3. The system does NOT hallucinate, does NOT repeat questions, does NOT show stale state in the side panel, does NOT fabricate citations, and does NOT show irrelevant cards.

4. Speed: every interaction completes within human-acceptable bounds (per-turn <8s, research <90s, dashboard load <3s).

These apply to all 4 personas. Each must be checked per persona.

### a. No repeated questions

Track every question text shown by Question Director throughout the onboarding flow. Maintain a Set. If any question text appears twice, FAIL with the duplicate text logged.

This is a real bug observed during development — the side panel showed "upcoming question" matching a question already answered twice. Must be regression-tested forever.

### b. Side panel sync with chat

When Question Director shows question N in the chat:
- The side panel's "current focus" or "next field" indicator must match field N within 2 seconds
- The side panel's filled-fields list must show all fields populated up to (but not including) field N

If side panel shows a field as "next" while chat is asking a different field — FAIL.

### c. No hallucination

Profile fields stored in `relocation_plans.profile_data` (read after each turn via the side panel DOM, not direct DB) must satisfy:
- Every field value either matches what user explicitly typed OR is a defensible inference
- Defensible inferences are tagged `confidence='inferred'` with reasonable basis (e.g., birth_year from "I'm 28")

- If a field appears that user never volunteered AND wasn't reasonably inferable — FAIL with the field key + value logged

## BUG-HANDLING PROTOCOL

If ANY assertion fails for ANY persona, this protocol is binding:

### 1. STOP

Do not proceed to remaining personas. Do not skip the failing assertion. Do not mark the test as "mostly passing".

### 2. Root-cause the bug — NOT a workaround

Find WHY the system produced wrong output:

- **Side panel out of sync**: trace the SSE event flow from api-server's chat orchestration through the frontend's event handler. Find where state diverges from Question Director's actual next-field decision.
- **Repeated question**: trace Question Director's missing-field calculation. Find why it returned a field key that was already populated in profile_data.
- **Hallucinated field**: trace Extractor agent output. Find why it returned a value the user never said. Tighten the extractor's system prompt or add a guard rail.
- **Wrong specialist dispatched**: trace Coordinator's dispatch logic. Find why it called posted-worker for Roselle (or whatever the mismatch is). Fix the trigger predicate.
- **Missing card**: trace shouldShow predicate. Find why it returned false. Fix the predicate logic.
- **Broken citation URL**: trace the specialist that produced it. Either the scraping logic returned a stale URL, or the LLM fabricated one. Either way, fix at source.

Read the actual code, the actual SSE messages (via DevTools Network tab or by adding temporary logging), the actual DB state (read via Supabase Studio, not direct DB writes from test). Don't guess. Don't assume.

### 3. Fix at the source — acceptable vs unacceptable fixes

**Acceptable fixes:**
- Correct logic in the agent (extractor, validator, question director, coordinator, specialist)
- Fix race condition in SSE event ordering
- Tighten an LLM prompt to prevent hallucination
- Fix shouldShow predicate to match the spec
- Fix citation extraction in a specialist

**NOT acceptable as "fixes":**
- Wrap the failing assertion in try/catch and log the failure
- Add `setTimeout` or `page.waitForTimeout(N)` to "wait for state to stabilize" (this papers over race conditions, doesn't fix them — `page.waitForResponse()` or `page.waitForSelector()` with a meaningful condition is acceptable; arbitrary sleeps are not)
- Mark the assertion as `.skip` or `.fixme`
- Lower a confidence threshold to make extraction "pass" with worse data
- Comment out the failing assertion
- Change the test to match the broken behavior
- Add a `try { ... } catch { /* known issue */ }` block
- Use `expect.soft()` to make a hard requirement soft

### 4. Re-verify the fix actually holds

After applying a fix:

1. Run the failing persona's test 3 TIMES IN A ROW
2. All 3 runs must pass (not 2 of 3, not "passed when I retried")
3. Then run all 4 personas back-to-back
4. All 4 must pass on the back-to-back run

If the fix passes once but fails on retry: it's not fixed. The bug is intermittent or the fix introduced a different issue. Continue investigation.
