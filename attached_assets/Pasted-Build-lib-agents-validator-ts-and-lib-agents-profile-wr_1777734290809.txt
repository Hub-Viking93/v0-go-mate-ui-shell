Validator (lib/agents/validator.ts):
Pure code, no LLM. Takes the raw value from Extractor and the field key. Runs through schema validation.

validate( rawValue: string | number | "yes" | "no" | null, fieldKey: AllFieldKey ): ValidationResult

type ValidationResult = | { valid: true, normalizedValue: any } | { valid: false, errorCode: string, retryHint: string }


Validation rules per field type, deterministic:
- yes/no fields: normalize "yeah"/"yep"/"sure"/"definitely"/"of course"/"absolutely" → "yes"; "nah"/"nope"/"no way"/"not really" → "no". If unclear → invalid.
- birth_year: accept 4-digit year between 1900 and current year minus 16.
- date fields (target_date, etc.): parse common date formats.
- currency fields (savings_available, monthly_budget, monthly_income, rental_budget_max): accept patterns like "5000 USD", "5000 SEK", "$5000", "€5000". Normalize to "5000 USD" format.
- country fields (citizenship, destination, current_location, prior_visa_country, vehicle_origin_country): use existing country-name-normalizer.ts to validate against country list. Accept synonyms ("USA"→"United States").
- city fields (target_city): leave as-is, just trim.
- numeric fields (children_count, posting_duration_months, commute_tolerance_minutes): parse as integer.
- percentage / months fields (income_history_months, posting_duration_months): parse with units, convert to base unit (months).
- enum fields (visa_role, purpose, settlement_reason, study_type, etc.): match against allowed values.
- string fields with no constraints: accept as-is, trim.

For "I'm 30 years old" given fieldKey="birth_year": this would already have been processed by the Extractor (which returns "1995" or similar). If somehow the Extractor returned the literal string, the Validator catches it because it's not a 4-digit year.

For "I am 30" given fieldKey="name": the Validator rejects this with errorCode="not_a_name", retryHint="That sounds like an age. What's your name?"
