# GoMate — Glossary

Definitions for all domain-specific terms used across GoMate documentation, code, and system docs.

If a term appears in `docs/build-protocol.md`, `docs/audit.md`, or any system doc, its canonical definition is here.

---

## A

### Arrival
The user action of confirming they have physically relocated to their destination. Triggered by `POST /api/settling-in/arrive`, which transitions the plan's `stage` from `complete` to `arrived`.

### Artifact (gate artifact)
A required output document produced after a phase gate passes. Gate artifacts are proof that the gate was completed correctly. The three mandatory phase artifacts are: `backend-acceptance-phase-N.md`, `frontend-wiring-report-phase-N.md`, `regression-report-phase-N.md`. See `docs/phase-implementation-protocol.md`.

---

## C

### Chat (pre-arrival)
The structured AI interview a user completes before arrival. Powered by GPT-4o via raw fetch to OpenRouter. Extracts profile fields from conversational answers and drives the state machine through `collecting → generating → complete`.

### Chat (post-arrival)
A context-aware assistant available after `plan.stage === 'arrived'`. Powered by the same `/api/chat` route but with a different system prompt that injects the user's settling-in task list. Can mark tasks complete via the marker protocol.

### Checklist (document checklist)
The list of documents the user needs to prepare for their relocation. Stored as `document_statuses` JSONB on `relocation_plans`. Shown on the `/documents` page.

### Checklist (requirements checklist)
The research-generated list of visa and administrative requirements for the user's destination. Generated by `lib/gomate/checklist-generator.ts` via Claude Sonnet + Firecrawl.

### Compliance Timeline
An ordered, deadline-aware view of the user's settling-in tasks that have legal or administrative deadlines. Rendered by `components/compliance-timeline.tsx`. Dismissals are v1 localStorage-only (no server persistence).

### `computeAvailableTasks()`
A function in `lib/gomate/settling-in-generator.ts` that computes which tasks in the DAG are currently unlocked (all dependencies satisfied). This function is WORKING and must not be changed without explicit instruction.

---

## D

### DAG (Directed Acyclic Graph)
The task dependency structure for settling-in tasks. Each task can have prerequisite tasks (dependencies). The graph must have no cycles — cycles cause permanent task deadlock. DAG cycle detection is added in Phase 2.

### Dependency (task dependency)
A relationship between two settling-in tasks where one task must be completed before another can be started. Stored as a `dependencies` JSONB array of task UUIDs on `settling_in_tasks`.

---

## F

### `fetchWithRetry()`
A shared utility function that wraps `fetch()` with exponential backoff retry logic and an `AbortController` timeout. Created in Phase 4. Located at `lib/gomate/fetch-with-retry.ts`. All external HTTP calls must use this or an equivalent timeout pattern.

---

## G

### Gate (phase gate)
One of the seven mandatory steps in the phase implementation protocol. Gates must be executed in exact order with no skips. Each gate requires artifacts as proof of completion. See `docs/phase-implementation-protocol.md`.

### Guide
An AI-assisted country guide generated for a user's destination. Stored in the `guides` table. Contains sections: visa, banking, cost of living, housing, culture, etc. A guide can be downloaded as a PDF from `/guides/[id]`.

### `getUserTier()`
A function in `lib/gomate/tier.ts` that reads the user's subscription tier from the `user_subscriptions` table. Returns `free | pro_single | pro_plus`. Every Pro+ gated API route must call this.

---

## I

### Interview
See **Chat (pre-arrival)**. The interview is the structured Q&A flow that builds the user's Profile.

---

## L

### Lock (plan lock)
The moment the user's pre-arrival interview is complete and confirmed. Locking triggers the research pipeline (visa research, cost of living, checklist generation) and transitions the plan to `stage = 'generating'` → `'complete'`.

---

## M

### Marker Protocol
The mechanism by which post-arrival chat marks tasks complete. The AI assistant emits `[TASK_DONE:exact task title]` in its response. The frontend parses this with regex and fires `PATCH /api/settling-in/[id]`. The format uses the task **title string**, not a UUID. This is intentional v1 design. Do not change the format without updating both the system prompt and the frontend parser.

### Migration
A numbered SQL file in `scripts/` that defines a database schema change. Files are numbered sequentially (001, 002, ...). Every column referenced by application code must have a corresponding migration. The next migration number is **015**. Migrations must use `ADD COLUMN IF NOT EXISTS` to be idempotent.

### Migration 004 gap
There is no `scripts/004_*.sql` file in the repository. The sequence jumps from 003 to 005. This is an inherited gap from early development. Do not create a 004 file.

---

## P

### Phase
A bounded unit of work defined in `docs/build-protocol.md`. Phases 0–5 cover the full v1 implementation. Each phase targets specific gap codes from `docs/audit.md`. Phases must be completed in order (0 → 1 → 2 → 3 → 4 → 5).

| Phase | Focus |
|---|---|
| Phase 0 | Schema integrity (migrations) — COMPLETE |
| Phase 1 | P0 security fixes |
| Phase 2 | Settling-in stage integrity |
| Phase 3 | Data integrity |
| Phase 4 | Reliability minimum |
| Phase 5 | UI integrity |

### Plan
A user's relocation scenario. Each plan has: a destination, a 65-field profile (stored as JSONB), a stage, a status, research results, and a task graph. Users can have multiple plans but only one active plan at a time. Plans are stored in the `relocation_plans` table.

### `plan.stage`
An enum field on `relocation_plans` that tracks the relocation lifecycle:
```
collecting → generating → complete → arrived
```
See **Stage**.

### `plan.status`
A separate field from `plan.stage`. Valid values: `active`, `archived`. An archived plan is one the user has put aside. Archived plans cannot be switched to and cannot have settling-in tasks added. `archived` is NOT a stage value.

### Profile
The structured 65+ field data object representing everything known about a user's relocation. Stored as JSONB in `relocation_plans.profile`. Defined and validated by `lib/gomate/profile-schema.ts` (Zod schema). The profile is built incrementally through the chat interview.

### Profile (user profile)
The row in the `profiles` table, which contains display information (name, avatar). Not to be confused with the 65-field plan profile above.

### Pro+ (tier)
The highest subscription tier (`pro_plus`). Grants access to settling-in features, compliance timeline, post-arrival chat, and why-it-matters enrichment. Checked via `getUserTier()` on every gated API route.

---

## R

### Research Pipeline
The set of AI + web scraping operations that run after a plan is locked: visa research, local requirements lookup, cost-of-living analysis, and checklist generation. Triggered by `POST /api/research/trigger`. Uses Firecrawl and OpenAI.

### RLS (Row-Level Security)
Supabase's PostgreSQL row-level security policies that restrict which rows each user can read or write. All tables use RLS. Users can only access their own data.

---

## S

### Session
An authenticated Supabase GoTrue session. Refreshed on every request by the middleware (`middleware.ts` + `lib/supabase/middleware.ts`). Without a valid session, all protected API routes return 401.

### Settling-In
The post-arrival feature that generates a personalised task graph to help the user complete administrative steps after relocating. Available only when `plan.stage === 'arrived'`. A Pro+ feature. Tasks are generated by Claude Sonnet 4 via `lib/gomate/settling-in-generator.ts`.

### Settling-In Task
A node in the settling-in task graph. Stored in `settling_in_tasks`. Has: title, description, deadline, dependencies, status (`pending | in_progress | completed | skipped`), and enrichment data (`why_it_matters`).

### Stage
The `stage` field on `relocation_plans`. An ordered progression with exactly four values:

| Stage | Meaning |
|---|---|
| `collecting` | User is in the pre-arrival chat interview |
| `generating` | Plan locked; research pipeline running |
| `complete` | Research done; pre-arrival dashboard active |
| `arrived` | User confirmed arrival; post-arrival mode active |

**`archived` is NOT a stage. It is a `plan.status` value.**

### State Machine
The interview state machine defined in `lib/gomate/state-machine.ts`. Manages transitions through the pre-arrival chat flow. Do not change without explicit instruction.

### System Prompt
The instructions injected into the AI chat context to define its persona, capabilities, and behaviour. Pre-arrival and post-arrival prompts are built by `lib/gomate/system-prompt.ts`.

---

## T

### Tier
The user's subscription level. Three valid values: `free`, `pro_single`, `pro_plus`. Stored in `user_subscriptions.tier`. Enforced server-side via `getUserTier()` from `lib/gomate/tier.ts`.

- `free` — chat interview + dashboard only
- `pro_single` — one-time lifetime purchase; unlocks pre-arrival features (visa research, guides, cost of living, booking, documents)
- `pro_plus` — recurring subscription; adds post-arrival mode, settling-in, compliance timeline, multiple plans

### `TIER_FEATURES`
A feature matrix object mapping tier levels to feature flags. Defined in `lib/gomate/tier.ts` (server) and duplicated in `components/tier-gate.tsx` (client). This duplication is a known gap (G-7.1-G). The server copy is authoritative.

---

## V

### V1
The first production-ready release of GoMate. Defined by the V1 invariants in `docs/audit.md` section 2. All P0 gaps must be fixed. All 6 V1 invariants (INV-S1–S4, INV-D1–D5, INV-F1–F4) must hold. Phases 0–5 deliver V1.

### V2
Everything deferred beyond Phase 5: Job System, Observability, Artifact System, Stripe payments, chat history persistence, password reset, visa system consolidation. Do not build V2 items during V1 phases.

---

## W

### Why-It-Matters
An AI-generated explanation of why a specific settling-in task is important for the user's situation. Fetched on demand via `GET /api/settling-in/[id]/why-it-matters`. Generated by `lib/gomate/settling-in-generator.ts`. This system is WORKING.
